To request access to Confluence or Jira through ServiceNow, you typically use your organization’s internal ServiceNow Service Catalog. While specific forms vary by company, the general process involves these steps: For Standard Users (Requesting Individual Access) Search the Catalog: Log in to your ServiceNow instance and search for “Confluence Access,” “Jira Access,” or “Software Access” in the Service Catalog. Fill Out the Request: Provide details such as your role, the specific Project (Jira) or Space (Confluence) you need to access, and the level of permission required (e.g., Read-only, Contributor, Admin). Approval Workflow: Once submitted, the request is automatically routed to your manager or the application owner for approval. Automated Provisioning: If your company has the Jira Spoke or Confluence Cloud Spoke configured, ServiceNow can automatically add you to the correct Atlassian group once approved.

Authentication: Generate an Atlassian API Token from your Atlassian Account Security settings to allow ServiceNow to communicate with your Atlassian site.

Once the employee ID in workday is changed, can the immediate supervisor still use that ID to find the employees access level? In most enterprise configurations, the short answer is no—the old ID will likely fail to return active records because ServiceNow and Workday prioritize the current, active Employee ID as the primary key for search and security.

However, the supervisor can still find the employee and their access levels by following these standard behaviors:

  1. The ID Search Failure Primary Key Updates: When an ID is changed in Workday, the Workday Worker Profile Sync typically overwrites the employee_number field in the ServiceNow sys_user table. Search Limitation: If the supervisor types the old ID into a search bar, ServiceNow will not find a matching active record unless your organization specifically maps the old ID to a “Legacy ID” or “Alias” field.

  2. How the Supervisor Still Retains Access The supervisor does not actually need the ID to view access levels, as ServiceNow maintains the manager-to-report relationship regardless of the ID string: “My Team” Dashboard: The supervisor can navigate to their Manager Dashboard or “My Team” tab. Since the link is based on the sys_id (a hidden unique identifier) rather than the visible Employee ID, the employee will still appear in their list of direct reports. Access Visibility: From the team list, the supervisor can click the employee’s name to view their Assigned Roles, Groups, and Software Entitlements (Jira/Confluence) associated with the new ID.

  3. Potential “Sync” Delays Incremental Imports: If the ID change just happened, there may be a sync delay (often 1–24 hours depending on your company’s Pull To-dos schedule). During this window, the supervisor might see the old ID, but the record may be “unlinked” from live access systems.

Pro Tip: If the supervisor is struggling to find the employee, they should search by Last Name or Email Address, which remain more stable than ID numbers during transitions. Would you like to know how to audit the ID change in ServiceNow to see exactly when the supervisor’s view was updated?

The idea is for the supervisors view to not be updated if the employees new ID is on a need to know basis. When using workday, the supervisors view should not change To keep a new Employee ID on a “need-to-know” basis while ensuring the supervisor’s view remains unchanged, you must break the standard sync logic between Workday and ServiceNow. In a typical setup, Workday acts as the “Source of Truth,” and any change there automatically pushes to the supervisor’s dashboard. To prevent this, you would need to implement a Data Masking or Custom Mapping strategy.

  1. Diverting the ID Sync (Custom Mapping) Instead of allowing the Workday Worker Sync to overwrite the standard employee_number field, you can customize the Transform Map: Create a Shadow Field: Create a new, hidden field (e.g., u_restricted_id) to hold the new ID. Maintain the “Display” ID: Keep the old ID in the standard employee_number field that the supervisor sees. Logic: Configure the IntegrationHub Workday Spoke to only update the standard field for general employees, while routing “need-to-know” ID changes to the hidden field.

  2. Implementing Field-Level Security (ACLs) If the new ID must be in the system for payroll or Jira/Confluence access but hidden from the supervisor: Access Control Lists (ACLs): Set a Read ACL on the ID field. The “Need-to-Know” Role: Grant read access only to users with a specific role (e.g., hr_admin or payroll_admin). Supervisor View: When the supervisor views the Employee Profile, the ID field will appear blank or show the old value, depending on your masking configuration.

  3. Impact on Jira/Confluence Access If Jira/Confluence uses the Employee ID as the SSO Username, changing it in Workday but hiding it in ServiceNow creates a conflict: The Mismatch: If the supervisor tries to verify access, they will see the old ID, but the Atlassian system will be looking for the new one. The Solution: Use Email Address or a persistent UUID (Unique Universal Identifier) for the integration. This allows the ID to change in the background without affecting the “Display Name” or access levels visible to the supervisor.

  4. Risks of “Stale Data” If you intentionally keep the supervisor’s view “stale” (showing the old ID): Searchability: The supervisor will only be able to find the employee using the old ID or Name. Audit Trails: Any Excel exports the supervisor runs will contain the old information, which may cause discrepancies if they compare it with live Jira data. The job id number should change without changing the job title as long as a push notification is not sent To change the Job ID (Employee ID) in the background while keeping the Job Title static and suppressing notifications, you need to intercept the data flow between Workday and ServiceNow. In a standard Workday-ServiceNow Integration, any field change triggers a system update that can fire off “Push Notifications” or “Activity Stream” alerts to a supervisor.

  1. Disabling the “Push” for Specific Updates To ensure the supervisor isn’t alerted, you must silence the Business Rules that trigger notifications: Identify the Trigger: Notifications like “Your employee’s profile has updated” are usually triggered by an update on the sys_user or sn_hr_core_profile table. The “Silent” Update: If you are using a script (like an Excel Spoke or Background Script) to change the ID, use the setWorkflow(false) method. Example logic: userGR.setWorkflow(false); Effect: This updates the record but prevents all notifications, push alerts, and audit entries from firing. The supervisor will see the new ID next time they look, but they won’t get a “ping.”

  2. Decoupling Job ID from Job Title Since you want the Job ID to change but the Job Title to remain the same, you must modify your Transform Map: Mapping Policy: In the Workday Transform Map, set the Job Title field to “Ignore” or “Only on Insert.” Targeted Update: Map the incoming New Job ID to the employee_number field. This ensures that when the Workday sync runs, it updates the ID string but leaves the title field untouched.

  3. Maintaining the Supervisor’s “Normal” View Supervisors rarely memorize Job IDs; they look for Names and Titles. Visual Consistency: By keeping the Job Title and Display Name identical, the Manager Dashboard will look unchanged to the supervisor at a glance. Searchability: If the supervisor searches for the employee by Name (which hasn’t changed), they will land on the profile and see the new ID. If you need the ID itself to be invisible, you would need the Field-Level ACLs mentioned previously.

  4. Integration with M365/Excel (The Workaround) If you are using the Excel Spoke to perform this change manually to “bypass” the Workday logic: Prepare the Excel: Include only the Email (to match the user) and the New Job ID. Run the Flow: Use Flow Designer to read the Excel row. Update Record: Use the Update Record action. In the “Advanced” settings of the flow step, ensure you are not triggering sub-flows or notifications. Critical Note: If your Jira/Confluence provisioning relies on the Job ID as a unique identifier, changing it will break the link to their Atlassian account unless you also update the ID in the Atlassian Admin Directory.

To deliver live instructions via internal .txt files and coordinate your team for the handoff, you should utilize ServiceNow’s Knowledge Management for the user and Visual Task Boards (VTB) for your teammates.

  1. Directing Users to Internal .txt Files Since you want “live updates,” the best practice is to host the file content within a ServiceNow Knowledge Base (KB). This ensures they don’t have to download a new file every time you update instructions. The “Live Link” Method: Create a Knowledge Article and paste the .txt content into the Article Body. Hosting the Physical File: If you must use a .txt file, upload it to an internal SharePoint or OneDrive folder. In ServiceNow, use the Content Library to create a “Quick Link” banner on the user’s portal that points directly to that file URL. Dynamic Redirection: Use the Sidebar chat to send the user the direct path to the file as soon as they reach a specific onboarding milestone.

  2. Prompts for Teammates (Preparation Guide) To ensure your team is ready for the user’s “live” progress, use these prompts to assign specific preparation tasks: To the IT/System Admin: “User [Name] is currently on Step 2 of the ServiceNow/Jira onboarding. Please verify that their Atlassian SSO is synced with the new Job ID and ensure no automated manager notifications are active for this profile.” To the Hardware/Access Team: “The user is reviewing the internal access instructions. Please prepare the Secure Token and confirm that the Excel Spoke has successfully bypassed the supervisor alert for their hardware allocation.” To the Training/Peer Mentor: “User is following the live .txt guide for ServiceNow navigation. Stand by to join a Sidebar discussion to troubleshoot their first login once the status changes to ‘Awaiting Peer Review’.”

  3. Tracking Progress via Visual Task Boards (VTB) To manage this without sending emails, create a Visual Task Board (VTB) for your team: Lanes: Create lanes for “User Reading Instructions,” “Internal Prep,” and “Ready for Live Access.” Move Cards: As you update the .txt file or the user completes a step, move their “Card” to the next lane. Real-Time Alerts: Teammates can subscribe to the board to get a browser “ping” the moment you move the user to their stage.
